# 🤝 貢獻指南

感謝你對 Auto-Fix Bot 項目的興趣！我們歡迎各種形式的貢獻。

## 🌟 貢獻方式

### 報告 Bug
如果你發現了 bug，請：
1. 檢查 [Issues](https://github.com/we-can-fix/slasolve/issues) 確認問題未被報告
2. 創建新的 Issue，包含：
   - 清晰的標題和描述
   - 重現步驟
   - 預期行為 vs 實際行為
   - 環境信息（OS、版本等）
   - 相關日誌或截圖

### 提出功能建議
我們歡迎新功能的建議：
1. 在 Issues 中創建 Feature Request
2. 說明功能的用途和價值
3. 提供使用場景示例
4. 如果可能，提供實現思路

### 提交代碼
1. Fork 這個倉庫
2. 創建你的特性分支 (`git checkout -b feature/AmazingFeature`)
3. 提交你的更改 (`git commit -m 'Add some AmazingFeature'`)
4. 推送到分支 (`git push origin feature/AmazingFeature`)
5. 開啟一個 Pull Request

### 改進文檔
- 修正拼寫或語法錯誤
- 改進現有文檔的清晰度
- 添加缺失的文檔
- 翻譯文檔到其他語言

## 📋 開發指南

### 環境設置

```bash
# Clone 倉庫
git clone https://github.com/we-can-fix/slasolve.git
cd slasolve

# 安裝依賴（如果有）
npm install

# 運行測試
npm test
```

### 代碼風格

- 遵循現有的代碼風格
- 使用有意義的變量和函數名稱
- 添加適當的註釋
- 保持代碼簡潔易讀

### 提交訊息規範

使用清晰的提交訊息：

```
類型: 簡短描述

詳細描述（可選）

相關 Issue: #123
```

類型：
- `feat`: 新功能
- `fix`: Bug 修復
- `docs`: 文檔更新
- `style`: 代碼格式（不影響功能）
- `refactor`: 重構
- `test`: 測試相關
- `chore`: 構建過程或輔助工具的變動

### Pull Request 檢查清單

在提交 PR 前，請確認：
- [ ] 代碼遵循項目風格
- [ ] 添加了必要的測試
- [ ] 所有測試通過
- [ ] 更新了相關文檔
- [ ] 提交訊息清晰明確
- [ ] PR 描述完整
